# 学生ゴルフ選手権 エントリーAPI (バックエンド・ポートフォリオ)

高トラフィックな予約システムを想定し、**「同時アクセス時の排他制御」**と**「テスト容易性を意識した設計」**に焦点を当てて開発した、Go言語によるREST APIサーバーです。

## 🎯 アピールポイント（工夫した点）

### 1. データベースレベルでの排他制御（オーバーブッキング防止）
定員10名の大会に数百人が同時にアクセスする状況を想定し、データの不整合を防ぐ工夫をしています。
Goのプログラム内で `SELECT` してから `INSERT` するのではなく、PostgreSQLのトランザクション内で以下のクエリを実行しています。
```sql
UPDATE tournaments 
SET current_entries = current_entries + 1 
WHERE id = $1 AND current_entries < capacity;
```
これにより、DBの行ロック機能を活用したアトミックな更新を実現し、レースコンディション（競合）を完全に防いでいます。

### 2. レイヤードアーキテクチャの採用と関心の分離
コードを以下の3層に分割し、インターフェース（`interface`）を用いて疎結合な設計にしました。
* **Handler（プレゼンテーション層）**: HTTPリクエストの受付とJSONの返却
* **Usecase（ビジネスロジック層）**: エントリーの業務ルール検証
* **Repository（データアクセス層）**: SQLの発行とトランザクション管理
これにより、Webフレームワークやデータベースの変更に強い、保守性の高い構造になっています。

### 3. モックを活用した単体テスト（Table-Driven Tests）
Usecase層のテストにおいて、本物のデータベースを起動するのではなく、Repositoryの「モック（偽物）」を依存性注入（DI）することで、外部環境に依存しない高速な単体テストを実現しています。Goの標準的な「テーブル駆動テスト」を採用し、正常系・異常系を網羅しています。

## 🛠 技術スタック
* **言語**: Go (1.22+)
* **データベース**: PostgreSQL 15
* **インフラ**: Docker / Docker Compose
* **アーキテクチャ**: レイヤードアーキテクチャ, Dependency Injection (DI)

---

## 🚀 環境構築と起動方法

### 1. データベースの起動
Dockerを使用してPostgreSQLを立ち上げます。初回起動時に自動的にテーブルが作成されます。
```bash
docker compose up -d
```

### 2. APIサーバーの起動
以下のコマンドでサーバーを起動します。
```bash
go run ./cmd/api/main.go
```
起動後、`http://localhost:8080/api/tournaments` にアクセスしてデータが取得できれば成功です。

---

## 🧪 テストの実行方法
モックを利用したUsecase層のビジネスロジックのテストを実行します。
```bash
go test -v ./internal/usecase
```
